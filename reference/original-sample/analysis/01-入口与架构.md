# 入口与架构

## 1. HTML 壳层

`index.html` 很薄，但非常关键：

- 引入 `https://appsforoffice.microsoft.com/lib/1/hosted/office.js`
- 删除 `history.pushState` / `history.replaceState`
- 把 React 应用挂到 `#react-container`
- 在 HTML 层处理 Office 快捷键，避免 SPA 路由和任务窗格可见性冲突

这里说明它的顶层设计是：

1. HTML 壳层负责 Office 宿主兼容
2. React 负责实际 UI、状态和业务逻辑

## 2. 启动链路

从当前样本看，前端启动顺序大致是：

1. Office taskpane 加载 `index.html`
2. HTML 壳层引入 Office.js
3. 加载主 bundle `index-CaYG1oEg.js`
4. 初始化 telemetry
5. 监听 `KEYBOARD_SHORTCUT_TRIGGERED`
6. `Office.onReady(...)` 记录宿主能力快照
7. React Router 挂载到 `#react-container`

## 3. 路由结构

当前样本可见路由如下：

- `/`
- `/auth/login`
- `/auth/callback`
- `/auth/3p`
- `/auth/3p/gateway`
- `/auth/3p/vertex`
- `/auth/3p/bedrock`
- `/onboarding`
- `/terms`

这套路由表明它的首屏不是固定聊天页，而是带完整认证、条款、引导分流的产品壳。

## 4. 根路由的 gating 逻辑

`gtn()` 是首页总入口，它会综合以下状态决定跳转：

- 是否已认证
- 是否走 bootstrap / entra 路线
- 是否接受 terms
- 是否完成 onboarding
- 是否启用 MCP / conductor 等 feature gate

换句话说，真正的“主界面”并不是一进入就展示，而是被多层前置状态门控包住。

## 5. React 应用之外的 Office 专属桥接

HTML 和主 bundle 之间有一个很明显的桥：

- HTML 壳层在 Office 快捷键触发时 `window.postMessage({ type: "KEYBOARD_SHORTCUT_TRIGGERED" ... })`
- React 侧监听这个消息，然后尝试聚焦输入框

这说明开发者有意识地把“宿主相关动作”和“应用内交互”拆成两层。

## 6. 遥测初始化

主 bundle 初始化了：

- `endpoint: "https://pivot.claude.ai"`
- `serviceName: "office-agent"`
- `resourceAttributes: { "git.sha": "8a3d43cefdceab518a6d097570376a752fe96819" }`

同时还会上报 Office 宿主能力：

- ExcelApi / PowerPointApi / WordApi 最大 requirement set
- keyboard shortcut 支持情况
- Excel notes / displayed cell props
- Word comments / tracked changes

这说明 telemetry 并不是后补，而是产品级基础设施。

## 7. 我对整体分层的判断

当前前端大致可以拆成五层：

1. 宿主接入层
   - `index.html`
   - Office.js
   - keyboard shortcut / visibility bridge
2. UI 路由层
   - React
   - React Router
   - 登录、引导、条款、主界面
3. 会话与配置层
   - profile store
   - bootstrap
   - feature flags / GrowthBook
4. 工具与能力层
   - Office structured tools
   - `execute_office_js`
   - MCP
   - conductor
5. 远端服务层
   - `pivot.claude.ai`
   - `claude.ai`
   - `api.anthropic.com`
   - `mcp-proxy.anthropic.com`
   - `bridge.claudeusercontent.com`

## 8. 架构上的重要含义

- 这不是一个纯前端壳，而是一个“本地宿主 + 远端 agent platform”的混合 runtime。
- 很多能力是在前端先做约束和编排，再交给后端或宿主执行。
- 因为是统一 Office 工程，所以 Excel/Word/PPT 的逆向结果彼此能互相解释。
